## Responsibilities

### What Maintainers Do

✅ **Code Review & Merging**
- Review pull requests within 72 hours (target)
- Ensure code quality, tests, and documentation
- Merge approved changes

✅ **Issue Management**
- Triage new issues
- Label and prioritize issues
- Create "good first issues" for new contributors
- Close stale or resolved issues

✅ **Community Management**
- Welcome new contributors
- Answer questions and provide guidance
- Foster positive, inclusive environment
- Recognize and thank contributors

✅ **Project Planning**
- Define roadmap and priorities
- Make architectural decisions
- Plan releases
- Maintain project documentation

✅ **Continuity**
- Prepare handover documentation
- Identify and train successor maintainers
- Keep knowledge documented

### Response Time Expectations

| Activity | Target Response Time |
|----------|---------------------|
| Security issues | 24 hours |
| Bug reports | 48 hours |
| Pull requests | 72 hours |
| Feature requests | 1 week |
| General questions | 72 hours |

*During academic breaks or exam periods, response times may be longer. Maintainers will communicate absences in advance.*

## Decision Making

### Day-to-Day Decisions
Any maintainer can make decisions on:
- Merging non-controversial PRs
- Triaging and labeling issues
- Minor documentation updates
- Bug fixes

### Major Decisions
Require discussion and consensus among all maintainers:
- Architecture changes
- Breaking changes
- Major feature additions
- Changes to contribution guidelines
- Project archival

**Process:**
1. Open a discussion issue
2. Allow 3-7 days for maintainer input
3. Reach consensus or majority vote
4. Document decision

### Conflict Resolution
If maintainers disagree:
1. Discuss in private maintainer channel
2. If unresolved, escalate to NIT Raipur OSS Core Team
3. Core team mediates and provides recommendation
4. If still unresolved, faculty trustees make final decision

## Becoming a Maintainer

Interested in becoming a maintainer?

### Criteria

We look for contributors who:
- ✅ Made **5+ significant contributions** to the project
- ✅ Demonstrate **deep understanding** of codebase and architecture
- ✅ Show **good judgment** in code reviews and discussions
- ✅ Help and mentor **other contributors**
- ✅ Communicate **effectively and respectfully**
- ✅ Commit to **6+ month term** (minimum)

### Process

1. **Self-nominate** or be nominated by existing maintainer
2. **Express interest** to current maintainers
3. **Discussion period** (1-2 weeks)
   - Existing maintainers review contributions
   - Consider capacity and project needs
4. **Vote** by current maintainers (unanimous for first few, majority later)
5. **Onboarding** (1-2 weeks)
   - Granted maintainer access gradually
   - Shadow existing maintainers
   - Learn internal processes
6. **Announcement** to community

### Time Commitment

Expected commitment:
- **Active development phase:** 5-10 hours/week
- **Maintenance phase:** 2-5 hours/week
- **During contribution drives:** 8-15 hours/week
- **Minimum:** Check GitHub 2-3 times/week

## Stepping Down

Maintainers can step down at any time for any reason.

### Process

1. **Notify** co-maintainers at least **30 days** in advance (if possible)
2. **Handover** responsibilities:
   - Document current work and status
   - Transfer access to services
   - Share tribal knowledge
3. **Identify successor** (if possible)
4. **Announce** transition to community
5. **Move to emeritus status** (remain available for questions)

### Reasons to Step Down

Valid reasons include:
- Graduation
- Academic workload (exams, projects)
- Personal circumstances
- Loss of interest (totally okay!)
- Moving to different role in organization

No judgment - we appreciate your service!
